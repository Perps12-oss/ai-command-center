# Phase 4 — Chat Workspace v1.5 Reliability Constitutional Pre-Flight

## Task Description

Begin Chat Workspace v1.5 Phase 4 by hardening the chat lifecycle against stale/out-of-order events and ensuring every `CHAT_ERROR` event carries an attributable `request_id`. The first increment is scoped to:

1. `ai_command_center/services/chat_handler_service.py` — move `request_id` generation earlier so error events (empty clipboard, empty prompt after context assembly) include the active request ID, and clean up pending state for early-error paths.
2. `ai_command_center/ui/app.py` — filter incoming `CHAT_CHUNK` events against `AppState.active_chat_request_id` instead of a local UI variable, strengthening request ownership and stale-chunk suppression.
3. `tests/test_chat_workspace_v15.py` / `tests/test_chat_handler_phase4.py` — add targeted regression tests for stale error suppression and error `request_id` attribution.

Larger async/timeout/fallback restructuring of `ChatHandlerService` is deferred to a later Phase 4 increment.

## Regression Risk

Reasoning:
1. No service-to-service calls introduced.
2. No contract or topic version changes.
3. Chat handler error paths are already guarded; we only add `request_id` attribution and cleanup.

5. AppState reducers already ignore stale terminal events; the chunk filter now aligns with the same `request_id` ownership.
